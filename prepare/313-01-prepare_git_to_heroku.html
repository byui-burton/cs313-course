Deploying Content to Heroku

Before starting these instructions, make sure you have Set up your Heroku application.

Background

The easiest way to deploy content to Heroku is to use Git. These steps will walk you through setting up Git and using it to deploy to your server.

In this course we assume you already have basic experience with Git from your earlier courses. If you do not, you will still be ok, but you will need to come up to speed on your own. Here is some helpful information:

  • An introductory tutorial on YouTube
  • The official documentation - (It's actually well-written and relatively easy to follow!)

Instructions

The following steps will help walk you through setting up Git on your computer and using it to connect to Heroku and deploy a Hello World page. Later we will also use get to push our source code to a public source control server (GitHub). This can be a little bit confusing at first because we will have three separate Git repositories:

  1. On your local computer - You have to commit changes here, before you can push them anywhere else.
  2. At Heroku - We push here to deploy our application.
  3. At GitHub - We push here to share our code with our classmates and the instructor.

In the previous steps when you issued the "git clone" command, this created a new Git repository on your local system. Then, when you added the "remote" for Heroku, this pointed it to a git repository on their system. Issuing the "git push heroku master" command essentially said, sync up the latest changes from my local system to Heroku.

Thus, when we want to make additional changes, we change the local copy, commit it to the local repository and then, push it to Heroku.

Deploying custom content to your site

  1. On your local computer, in a terminal window, go to the directory where you cloned the repository earlier.
  2. Change into the "web" subdirectory. This is the root directory of your website. There is an index.php file already in this directory. This is the content that was loaded when you went to your page in a browser.
  3. Create your own hello.html page (using any editor of your choice) that displays the text Hello World.
  4. In your terminal window, type "git status" to show the changes have been made. In this case, it should show that you have created a new hello.html page (that is currently not-tracked).
  5. Add your hello.html page to be tracked by Git (git add hello.html).
  6. Now, type "git status" again to see the changes that are ready to commit.
  7. Commit your changes to your local Git repository (git commit -m "Adding my hello world page")
    8. If you forget to include a message, Git will set you to its default text editor to create one. This might be VIM, which you may or may not be familiar with. If it is and you "just want to get out of it!" hit ESC, the type ":q!" to quit without saving changes. You will not have added a message, so your commit will not have gone through, but at least you have gotten out alive! :-).
  8. Now, your changes are in your local Git repository, but we still need to push them to Heroku. Do this by "pushing" them with the command "git push heroku master".
  9. When you push code to Heroku, it automatically kicks off a build that will do any actions necessary to deploy your site. You will set an output of these actions in your terminal window.
  10. Verify that your new hello world page is up. You can run the command "heroku open" to go to the default page, then add "hello.html" to the end of the URL. (It should be something like: https://fierce-hamlet-90863.herokuapp.com/hello.html).
  11. Celebrate. You have just pushed your first content to your site! Throughout the semester you will repeat this process many times. Put new content into this directory, "git add", "git commit" and "git push".
    12. You are more than welcome to use any GUI tools for Git that you would like. I find them to be helpful when everything is going well, but when something weird happens, going to the terminal window and running a "git status" command usually helps me track down problems better than anything else.
